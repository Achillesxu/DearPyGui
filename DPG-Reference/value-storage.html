Dear PyGui

Widget Values

In Dear PyGui, a widget's underlying value is treated differently than in most other UI libraries. In most UI libraries, the widget owns the value it controls. This is not the case with Dear PyGui. Widgets share values that are stored in a central location. As you will see, this provides more flexibility than the traditional encapsulated values.

Value Storage System

Values are stored in the value storage system. This system acts as a central location with the following responsibilities:

  • Ensure proper cleanup when a value is no longer needed (uses reference counting)
  • Ensure type compatibility.
  • Minimize conversions between C and Python types.

Every value entry must have a unique ID. Once a value's type is set, it can not be changed.

Adding Values to Value Storage System

There are 2 ways in which you can add to the value storage system.

  • Create a widget that has an underlying value.
  • Using the add_value command.

When you create a widget with an underlying value, for example a input_float. By default, a value is added to the value system with its unique ID being the same as the widget itself. The type is also decided by the addition of this new widget. In this example, the type is a float.

If you would like to override the unique ID of the value, you can use the source keyword. This is useful if you would like multiple widgets to control the same underlying value: 

add_input_float("Widget 1", source="value1")
add_drag_float("Widget 2", source="value1")

In the above example, changing one widget will also change the other.

If you try to add a widget with an incompatible existing type, the value of the widget will fallback on a "common" value in the storage system.

Type Compatibility

Widgets that have compatible underlying types, can share the same value. For compatibility either of these must apply:

  1. Type need to be the same as the one in the value system.
  2. Type need to be a smaller sized array of the one in the value system

The first case is obvious, however the second leads to a few possibilities. A input_float can share a value with a drag_float4, assuming the underlying value is a float4. This is handled automatically if the drag_float4 was created first (because the type is registered as a float4). However, the drag_float4 would not be able to share the value if the input_float was created first since the type would be registered as a float, breaking compatibility Rule 2.

The solution is to add the value before creating the widgets. For example: 

add_value("value1", [0, 0, 0, 0]) # preregister value, must decrement ref count later
add_input_float("Widget 1", source="value1")
add_drag_float4("Widget 2", source="value1")

Reference Counts

Values are kept "alive" so long as their reference count does not reach 0.

The following procedures increment the reference count of their value by 1:

  • Creating a value with add_value
  • Adding a widget
  • Calling incref_value

The following procedures decrement the reference count of their value by 1:

  • Deleting a widget
  • Calling decref_value
